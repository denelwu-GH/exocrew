# ExoCrew

## AI 时代的交付外骨骼

**装上你还没有的那支团队。**

ExoCrew 不是代码生成器，也不是“一键做完企业系统”的魔法工具。它把产品、研发、测试、运维四类角色的检查视角，安装进你的 AI 编程工作流。

它的目标不是让一个人冒充一家公司，而是让一个人在借助 AI 推进真实产品时，知道需求还缺什么、代码应该守住什么、测试必须证明什么，以及上线前什么时候必须停下来。

ExoCrew 是一套行业无关、技术栈无关的通用交付框架，不限定教育、电商或任何特定业务领域。它适用于企业后台、SaaS、内部工具、运营管理系统，以及其他需要长期维护、持续测试和安全上线的软件项目。

## 背景背书

我不是程序员。

但在约四个月里，我借助 Codex，独立推进出了一套覆盖 25 个运营入口、401 个接口操作、125 个数据模型的复杂企业业务后台；Git 首尾留痕跨度为 139 个自然日。

**这不是 Demo，也不是比赛原型或停留在文档里的参考架构。** 这套系统已经完成真实生产上线，正在企业端承载日常运营、真实数据治理和企业业务信息推送。ExoCrew 提炼的不是“怎样快速做出一个能演示的页面”，而是怎样让一个持续运行的真实系统理解业务边界、遵守架构约束、验证每次改动、安全处理数据，并知道什么时候不能上线。

这段经历留下了约 24.5 万行运行时代码、11.6 万行测试源码、1,800+ 次主线提交，也沉淀出 90 条硬门禁、249 项架构决策、64 份操作手册和数百份发布与故障复盘。

现在，我把这些经验提炼成 ExoCrew。

它不是一份万能提示词。它是一套可以安装给 AI 编程代理的交付纪律：帮助你把模糊想法变成需求，把需求变成可维护的工程，把工程变成经过验证、能够发布和回滚的真实产品。

你不需要突然成为程序员。

你需要的是产品经理的边界意识、工程师的结构、测试人员的怀疑精神，以及运维人员对风险的敬畏。

ExoCrew 把这支你还没有的团队，装进你的 AI 工作流里。

一个人，也可以开始推进真正复杂的业务系统。

绑定一个已提交版本的只读审计显示：

- 139 个自然日 Git 留痕、主线 1,869 次提交
- 25 个运营功能入口、401 个显式 HTTP 操作
- 125 个数据模型、145 份数据库迁移
- 约 244,799 行运行时代码
- 466 个测试文件、约 116,871 行测试源码、9,264 个断言调用
- 有留痕的完整 E2E 运行记录：88 项通过、22 项跳过、0 项失败，随后整套发布前门禁通过
- 90 条硬门禁、249 项架构决策、64 份正式操作手册、17 个领域 Skill、573 份发布或修复复盘

这些数字和历史执行记录证明的是 ExoCrew 的生产提炼来源，不代表本仓库包含原业务系统，也不承诺测试永远不会失败，更不代表 573 份复盘等于 573 次生产上线。完整边界见 [证据与宣传口径](docs/EVIDENCE.md)。

![一个人借助 Codex 与传统交付团队的工作量对比](docs/assets/effort-comparison.zh-CN.svg)

透明重建模型使用两套独立团队配置与合理交付周期交叉测算：基础完整重建约需 **1,320–2,160 人天**，相当于 **17–27 名产品、研发、测试和运维角色工作同样约四个月**。若再计入业务探索、第三方平台踩坑、生产故障、数据治理和历史兼容，扩展情景约为 **2,000–3,000 人天 / 25–38 人同期工作面**。完整公式和边界见[测算依据](docs/EFFORT_MODEL.md)。

## 你获得什么

| Skill | 角色 | 给普通人的价值 |
|---|---|---|
| `exocrew-delivery` | 交付负责人 | 分清任务、证据、审批、停止条件和收尾标准 |
| `product-brief` | 产品 | 把模糊想法变成用户、价值、边界和验收标准 |
| `engineering-guardrails` | 研发 | 约束架构、契约、单一真值和异常可见性 |
| `test-evidence` | 测试 | 按风险设计验证，防止“看起来通过”的假绿 |
| `safe-operations` | 运维 | 约束数据变更、迁移、发布、回滚和上线后验证 |

## 安装 Codex Plugin

```bash
codex plugin marketplace add denelwu-GH/exocrew
codex plugin add exocrew@exocrew
```

安装后新建一个 Codex 任务，让新 Skills 被完整加载。

## 单独安装 Skills

克隆仓库后先预览：

```bash
node plugins/exocrew/scripts/install-skills.mjs
```

确认后执行：

```bash
node plugins/exocrew/scripts/install-skills.mjs --apply
```

默认安装到 `${CODEX_HOME}/skills`；未设置 `CODEX_HOME` 时使用 `${HOME}/.codex/skills`。已有 Skill 默认不覆盖；显式使用 `--force` 时会先备份。

## 为项目安装治理骨架

```bash
node plugins/exocrew/scripts/bootstrap-project.mjs --target ./my-project
node plugins/exocrew/scripts/bootstrap-project.mjs --target ./my-project --apply
```

第一条命令只预览，第二条才写入。starter 会创建唯一决策源、任务态、硬约束、索引，以及 worklog、决策、发布和经验模板。

## 利他价值

ExoCrew 把真实复杂项目里反复付过的学习成本，变成可以复用的提问、门禁、模板和停止条件：

- 产品上，减少需求模糊、状态遗漏和异常边界返工。
- 研发上，减少 AI 代码堆叠、契约漂移和重复真值。
- 测试上，减少弱断言、漏测关键链路和 scaffold 冒充验收。
- 运维上，减少无 dry-run 写入、迁移漂移、无回滚发布和口头收尾。

目前没有公开“提速多少倍、质量提升多少、少踩多少坑”的百分比，因为还没有完成安装前后的对照实验。项目提供 [30 个配对任务的 benchmark](docs/BENCHMARK.md)，未来只发布有样本量与置信区间的数据。

重建工作量描述的是提炼来源本身，不等于其他人安装 ExoCrew 后必然获得相同比例的提速。ExoCrew 的实际采用收益仍需要配对 Benchmark 验证。

## 结果畅想

普通人不会因为安装一个 Skill 就突然变成四类专家。

但他可以获得四类专家的检查视角：像产品经理一样问边界，像工程师一样守结构，像测试人员一样怀疑结果，像运维人员一样敬畏上线。

这才是超级个体真正可持续的来源。
